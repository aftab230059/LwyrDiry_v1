8 · Mobile Strategy & Project Plan

Q1 · Most users are on mobile — do I need to design again? Isn't this already responsive?

Short answer: you do NOT need a redesign — but it is not yet truly responsive. The current screens are desktop-first: a fixed ~1180px shell with a permanent left calendar column beside the case list, wide data tables (Pending / All Cases), and edit panels laid out in 2-column grids. On a phone these overflow and shrink below comfortable tap sizes.

Keep everything that defines the product — the visual language, type, colours, components and copy. What's missing is a responsive layer: breakpoints and a few mobile-specific patterns so the same design reflows to one column.

✓ Reuse design system ✓ Add breakpoints ✗ No full redesign
Q2 · How should I approach the project now? A breakdown to-do plan.
Phase 0Lock the spec. Sign off this document; freeze schema & the missing-date / reminder rules. Create the Supabase project (Postgres + Auth + Storage).
Phase 1Data & auth. Build the tables, foreign keys, indexes and Row-Level Security; implement register / login + OTP and password reset.
Phase 2Wire the existing UI to the API. Replace the current localStorage demo store with real endpoints: case CRUD, parties & contacts, scheduling / forward, personal reminders.
Phase 3Responsive / mobile pass (see Q3) — the single biggest UX win for your audience.
Phase 4Notifications & the reminder job. Scheduled missing-date scan (§7.2), push / SMS / in-app delivery, notifications log.
Phase 5Documents, payments, communication. File upload + signed URLs, fee tracking diagram, WhatsApp/Bangla client updates.
Phase 6Ship. PWA + Android wrapper (Capacitor/TWA), Bangla i18n, audit log, encrypted backups, then a pilot with 3–5 advocates and iterate.
Q3 · Making the mobile view perfect — concrete steps
Shell: breakpoints at 1024 / 720 / 480px; below ~720px collapse the two-column layout to a single stacked column.
Calendar: move the sidebar calendar into a slide-up sheet / drawer triggered from a top bar; keep the new month + year pickers.
Navigation: add a bottom nav (Diary · Pending · All cases · New) with ≥ 44px targets and safe-area insets.
Lists: the wide Pending / All-Cases tables become stacked cards on phones, full tables on desktop.
Case tile: drop the left case-no panel inline; the edit grids collapse to one column with a sticky Save bar.
Inputs: ≥ 16px font on inputs (stops iOS auto-zoom); full-width date/fee fields; native date pickers.
Performance: test on low-end Android & slow networks; lazy-load case files; cache for offline read (PWA).
Touch: swipe to expand a case, pull-to-refresh the day list, larger hit areas on Sunani/Todbir toggles.
Next step (done): the four mobile sample views — Day view, Calendar sheet, Pending cards, Case edit — are built in the “Mobile Views” file as the responsive build targets. Same components, added breakpoints; no visual redesign. The live Diary, All Cases and Pending Dates screens now ship these breakpoints \u2014 single-column shell, full-width case tiles, stacked calendar & flexible search below ~880px (Create Case already used auto-fit grids).
Q4 · Backend, database, hosting, version control, mobile — untangled (solo dev, 3–4 day deadline)

It feels like five problems; it is really three free services wired together once. Plain-English map first, then a day-by-day plan.

Your whole stack = GitHub (code) + Supabase (database + backend + auth) + Vercel (hosting). Three accounts, all free to start.
Version controlGit + GitHubSave points for your code — undo anything, back it up. One-time setup.
DatabaseSupabase (Postgres)Where every case & client lives. Fully managed — you never run a server.
Backend / APISupabase auto-API + RLSSupabase turns your tables into a secure API; Row-Level Security keeps each advocate private. Very little custom code.
Auth (login)Supabase AuthRegister / login + OTP, built in. No password handling of your own.
HostingVercelWhere the app runs online; redeploys automatically on every GitHub push. Free tier.
